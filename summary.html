User Research

User Interviews

User interviews are an important part of the user research process because they help you target the specific features hiring managers or potential viewers are interested in seeing. The types of questions asked during a user interview should be professional if talking to a hiring manager but informal enough to get their personal opinions about what THEY are looking for in an online porfolio.

Competitor Analysis

A competitor analysis is looking at other portfolios or websites that might have similar intentions as your own. By conducting competitor analyses, you can get a clearer picture about the content that your competitors have on their websites. This is a great way to draw inspiration and ideas for what to include in your own portfolio/website.

View Doc

Feature Value Matrix

A feature value matrix is a technique where you incorporate and map values/numbers to certain types of features you want to implement in your website/portfolio. This helps clarify which features are worth implementing and are the most crucial/important to include. For my feature matrix, I chose to include the features that were mentioned to me in the User Interviews I conducted. These two User Research methods/tactics go hand in hand.

View Doc

Feature Priortization

Through the feature value matrix, prioritizing features was as simple as seeing which values came out to be more important! When building an interface, it is important to know which features are the most important and valuable to implement because writing code is a costly, time consuming effort. Being efficient and resourceful can not only save time and money, but it can also make the interface less bombarded with information that might be redundant or useless.

User Testing

User Stories

User Stories are a great way to verify if your features are implemented exactly the way you want them to be. Often times, implemented features end up behaving in ways different from what the developer intended. User Stories allow the developer to put themselves in the shoes of their possible users and see if their features would behave as expected.

View Doc

Cognitive Walkthroughs

A cognitive walkthrough is imperative to verifying your features are actually behaving correctly with real users. This is different from user stories in that you are actually walking through your website instead of making up use cases. While conducting these cognitive walkthroughs, I didn't encounter any problems with the users. It was actually a fun and rewarding experience to see how my features were interpreted by the user.

User Test

I chose to use the click tests and the question tests because they were very simple to implement yet very effective in getting the answers I was looking for. The hardest part about using these test cases was actually getting a good sample size of people to take your tests. It was hard to get a variety of people besides students and peers to look at my tests. Hopefully as I work on the portfolio, I'll have access to a wider variety of people to send my tests to!

Summary Of Findings

The user research that went into building my portfolio helped me narrow down the features and styling that I wanted to implement and design. I felt that the competitor analysis was very beneficial because it helped me draw inspiration from great designers and guided me to creating an asthetically pleasing portfolio. The user interviews helped me understand what exactly hiring managers and peers were looking for in an online portfolio. With this information, I was more than ready to code my portfolio in an efficient and iterative manner.

The user testing helped me verify that my ideas were being interpreted by actual users the way I wanted my features to be interpreted. By conducting cognitive walkthroughs, I saw in real time how users interacted with my portfolio and got feedback on what to improve on in my portfolio as well as which design choices worked and which didn't.

After conducting my user research and user testing, I had a better understanding of what worked in my portfolio and what didn't. The whole process of doing the research first before writing any code definitely made the process more structured and organized. After testing, changes that needed to be made were easier to find and conduct, thus making the entire process an effective iterative process.

Incorporation Of Findings

In my findings, a lot of users told me that my design was very boring or too minimal. It was also mentioned to me that I needed to add a resume link or page. In my contact me page, one of the TA's mentioned that my email link should be a mail-to link. These are a few of the changes that I need to implement in my portfolio.

In the future, I plan on changing up the design to make it less boring but not too drastically. I do like the simple design of my current portfolio and my utiization of Text Based design. I do plan to add a link to my resume or implement it similar to other portfolios and have my resume embedded within the portfolio. Thanks to this class, this porfilio will be with me for the rest of my software engineer career.
